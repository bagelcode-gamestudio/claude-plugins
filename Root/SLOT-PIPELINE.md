# Slot Development Pipeline - 슬롯 개발 파이프라인 전환

## 비전

NodeCanvas 비주얼 FSM 기반의 슬롯 개발을 **코드 기반**으로 점진적 전환하여,
AI 바이브코딩 + QA 자동화 + 문서 자동생성이 가능한 개발 파이프라인을 구축한다.

> 기존 프로세스를 존중하며, 하나씩 단계적으로 진행한다.

---

## 1. 궁극적 목표

### 1-1. QA 자동화

- 버튼, 팝업, 상호작용 테스트를 코드로 자동화
- 각 상태/전환에 테스트 포인트 노출
- 슬롯별 자동 TC(Test Case) 실행

### 1-2. 바이브코딩으로 신규 슬롯 제작

- AI가 코드만으로 슬롯 로직 작성 가능
- 코드 템플릿 + AI 지시 = 빠른 슬롯 프로토타이핑
- NodeCanvas .asset (AI 읽기 불가) → .cs 코드 (AI 읽기/수정 가능)

### 1-3. 슬롯별 컨텍스트 문서

각 슬롯에 대해 3종 문서 유지:

| 문서 | 대상 독자 | 내용 |
|------|-----------|------|
| 기획 문서 | 기획자, PM | 게임 개요, 피처, 보너스 흐름, 수학 모델 |
| 개발 문서 | 개발 에이전트 | 서버/클라 구조, Blackboard, API, 코드 주석 |
| QA 문서 | QA, 자동화 | TC 목록, 엣지 케이스, 회귀 테스트 시나리오 |

---

## 2. 변환 동기

1. **NodeCanvas 의존성 제거** - 서드파티 라이브러리 탈피
2. **확장성/자동화** - 356개 게임으로 확장 시 코드 생성/템플릿 활용
3. **성능** - NodeCanvas 런타임 오버헤드 제거
4. **유지보수/가독성** - 코드 리뷰, diff, AI 분석 가능

## 3. 게임 제작 프로세스 & 에이전트 역할

> 기존 프로세스를 먼저 이해하고 문서화한 후, 점진적으로 개선한다.

### 3-1. 4대 에이전트 역할

#### 기획자 (Planner Agent)

기획 문서를 받아서 **개발 에이전트들이 작업할 수 있는 형태**로 변환하는 역할.
실제 게임회사 기획자의 보조 역할.

| 항목 | 내용 |
|------|------|
| 입력 | 지정 양식의 기획 문서 |
| 출력 | 개발용 기획 문서 (에이전트가 읽고 작업 가능한 형태) |
| 핵심 역량 | 게임 기획의 세세한 부분까지 이해 (연출 방향, 시퀀스, 진행 흐름) |
| 모르면 | **반드시 사용자에게 질문** (추측 금지) |
| 상호작용 | 메쓰가이, 서버개발자, 클라개발자 모두에게 기획 전달 |

#### 메쓰가이 (Math Guy Agent)

기획을 기반으로 **게임의 수학 모델(Math)**을 설계하고 구현하는 역할.

| 항목 | 내용 |
|------|------|
| 입력 | 기획자의 기획 문서 |
| 출력 | 메쓰 문서 (릴셋, 확률, 보너스 구조, RTP 등) |
| 핵심 역량 | 릴셋 설계, 확률 계산, 윈페이 구조, 보너스 진입 확률, RTP 산출 |
| 문서 내용 | 서버 Value 파일 형태 + 유저 경험 흐름 (어떤 보너스에서 어떤 경험) |
| 상호작용 | **전 에이전트** (기획↔확률 조율, 서버↔수치 구현, 클라↔연출 확률) |

> 메쓰 문서 형태는 추후 정의 예정

#### 서버개발자 (Server Dev Agent)

games 레포에 게임을 추가하고 **서버 게임 로직을 구현**하는 역할.

| 항목 | 내용 |
|------|------|
| 입력 | 기획 문서 + 메쓰 문서 |
| 출력 | games/src/slot_game/{id}.ts (게임 로직 + 수치 구현) |
| 핵심 역량 | 게임 로직 구현, 확률/수치 적용, API 리스폰스 설계 |
| 상호작용 | **전 에이전트** (클라에서 리스폰스 문의, 수정 요청 등 대응) |

#### 클라개발자 (Client Dev Agent)

Unity 게임 컨텐츠 로직을 구현하는 역할. **사용자의 작업량을 최소화**하는 것이 목표.

| 항목 | 내용 |
|------|------|
| 입력 | 기획 문서 + 서버 리스폰스 구조 |
| 출력 | 스키마 구조, 피처 컨트롤러, 유틸리티, 연출 로직 |
| 핵심 역량 | Unity C#, 하이라키 설계, 스키마, FeatureController, 연출 적용 |
| 사용자 협업 | 하이라키 구조 같이 설계, 가능한 부분 직접 구현, 사용자 수작업 최소화 |
| 한계 | Unity 엔진 특성상 사람 손이 가장 많이 닿는 영역 |

### 3-2. 에이전트 상호작용 흐름

```
                    ┌─────────┐
                    │  기획자  │
                    │ Planner │
                    └────┬────┘
                         │ 기획 문서
                    ┌────▼────┐
                    │ 메쓰가이 │
                    │Math Guy │
                    └────┬────┘
                         │ 메쓰 문서
                ┌────────┴────────┐
                │                 │
           ┌────▼────┐      ┌────▼────┐
           │서버개발자│      │클라개발자│
           │Server   │◄────►│Client   │
           └─────────┘      └─────────┘
              리스폰스 구조 공유
              수정 요청/반영

※ 전 에이전트 간 양방향 상호작용 가능
```

### 3-3. 기존 게임 제작 흐름 (AS-IS)

```
기획서 → 메쓰 → 서버 게임 로직 → 클라이언트 FSM/BT 구성 → 피처 스크립트 → QA
```

| 단계 | 도구 | 산출물 |
|------|------|--------|
| 기획 | 기획서 (외부) | 게임 스펙 |
| 메쓰 | 스프레드시트 등 | 릴셋, 확률표, RTP |
| 서버 | TypeScript | games/src/slot_game/{id}.ts |
| 클라이언트 FSM | NodeCanvas 에디터 | .asset 파일 (FSM/BT) |
| 피처 스크립트 | Unity C# | FeatureController, SymbolBehaviour 등 |
| QA | 수동 테스트 | - |

### 3-2. 현재 규모

| 항목 | 수량 |
|------|------|
| FSM 에셋 | 103개 |
| BT 에셋 | 134개 |
| 구현된 게임 | 5개 (BBD, GDD, VP, RB, FG) |
| Action 클래스 | 635개 |
| Condition 클래스 | 74개 |

### 3-3. 유지할 것 vs 바꿀 것

| 유지 | 바꿀 것 |
|------|---------|
| 서버 게임 로직 구조 | NodeCanvas FSM/BT → 코드 기반 FSM |
| Blackboard 데이터 모델 (경로 체계) | .asset YAML → .cs 코드 |
| FeatureController 패턴 | 비주얼 에디터 의존 → 코드 에디터 |
| 기존 Action/Condition 기능 | QA: 수동 → 자동 |
| 서버-클라 통신 구조 | 문서: 없음/흩어짐 → 체계적 MD |

## 4. 목표 프로세스 (TO-BE)

```
기획서 ─→ 기획자 ─→ 메쓰가이 ─→ 서버개발자 ─→ 클라개발자 ─→ 자동 QA
  │     에이전트    에이전트     에이전트      에이전트     자동화
  │        │          │           │             │           │
  │     기획 MD     메쓰 MD    서버 로직     코드 FSM    TC 자동실행
  │                                          (.cs)
  └──────────────────────────────────────────────────────→ 3종 문서 자동생성
```

| 단계 | AS-IS | TO-BE | 기대 효과 |
|------|-------|-------|-----------|
| 기획 정리 | 수동 해석 | 기획자 에이전트 | 구조화된 기획 MD |
| 메쓰 | 스프레드시트 | 메쓰가이 에이전트 | 릴셋/확률/RTP 자동 산출 |
| 서버 | 수동 코딩 | 서버개발자 에이전트 | 기획+메쓰 기반 로직 생성 |
| 클라 FSM | NodeCanvas .asset | 클라개발자 + 코드 FSM | AI가 직접 작성/수정 가능 |
| QA | 수동 테스트 | 자동화 TC | 회귀 테스트 자동 실행 |
| 문서 | 없음/흩어짐 | 3종 MD 자동생성 | 기획/개발/QA 컨텍스트 체계화 |
