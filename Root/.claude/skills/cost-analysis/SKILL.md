# 비용 분석 스킬 (Cost Analysis)

경로/대안을 평가할 때 단일 지표가 아닌 **다차원 비용 모델**을 적용합니다.

> **💡 연구 기반**: Transition에 비용/위험을 부여하면 더 현실적이고 전략적인 판단이 가능합니다.

---

## 언제 사용하나요?

- architect 설계 완료 시
- 2개 이상의 대안 비교 시
- critic이 리스크 분석할 때
- 트레이드오프 분석이 필요할 때
- 경로 선택이 필요할 때

---

## 비용 요소 (Cost Factors)

| 요소 | 설명 | 스케일 | 가중치 |
|------|------|--------|--------|
| **effort** | 구현 노력 | 1-5 | 1.0 |
| **risk** | 실패/장애 위험 | 1-5 | 1.5 |
| **uncertainty** | 불확실성 (알 수 없는 요소) | 1-5 | 1.2 |
| **reversibility** | 롤백 용이성 (높을수록 좋음) | 1-5 | -1.0 |
| **resource** | 필요 리소스 (인력, 도구) | 1-5 | 0.8 |

### 요소 설명

| 요소 | 1 (낮음) | 3 (중간) | 5 (높음) |
|------|----------|----------|----------|
| effort | 몇 줄 수정 | 새 모듈 추가 | 대규모 리팩토링 |
| risk | 로컬 영향 | 연관 모듈 영향 | 시스템 전체 영향 |
| uncertainty | 검증된 패턴 | 일부 미지 영역 | 완전히 새로운 시도 |
| reversibility | git revert 가능 | 마이그레이션 필요 | 롤백 불가 |
| resource | 현재 역량 내 | 학습 필요 | 외부 도움 필요 |

---

## 효율 비용 공식 (c_eff)

```
c_eff = (effort × 1.0) + (risk × 1.5) + (uncertainty × 1.2)
        - (reversibility × 1.0) - available_resource
```

### 해석

- c_eff 값이 **낮을수록** 좋은 경로
- reversibility가 높으면 비용 감소 (실패해도 돌아갈 수 있음)
- risk와 uncertainty에 높은 가중치 (실패 비용이 크므로)

### 가중치 조정 가이드

| 상황 | 조정 |
|------|------|
| 초기 프로토타입 | risk 가중치 ↓ |
| 프로덕션 배포 | risk 가중치 ↑ |
| 시간 압박 | effort 가중치 ↑ |
| 실험적 기능 | uncertainty 가중치 ↓ |

## 출력 형식 (필수)

```markdown
## 비용 분석

### 분석 대상
- 결정 사항: [무엇을 결정하는가]
- 대안 수: [N개]

### 비용 평가 테이블

| 대안 | effort | risk | uncertainty | reversibility | resource | c_eff |
|------|--------|------|-------------|---------------|----------|-------|
| A [S] | 2     | 2    | 1           | 5             | 2        | 1.7   |
| B [B] | 3     | 2    | 2           | 4             | 2        | 4.9   |
| C [I] | 4     | 3    | 3           | 2             | 3        | 9.1   |

### 분석 결과

**최소 비용 경로:** A (c_eff = 1.7)

**c_eff 해석:**
| 대안 | c_eff | 해석 |
|------|-------|------|
| A    | 1.7   | 저비용, 안전 - 진행 권장 |
| B    | 4.9   | 중간 비용 - 검토 후 진행 |
| C    | 9.1   | 고비용, 주의 - 대안 탐색 권장 |

**권장:** [권장 대안과 이유]
```

---

## c_eff 임계값

| c_eff 범위 | 해석 | 권장 행동 |
|-----------|------|----------|
| **0-4** | 저비용, 안전 | 진행 권장 |
| **5-8** | 중간 비용 | 검토 후 진행 |
| **9-12** | 고비용, 주의 | 대안 탐색 권장 |
| **13+** | 매우 고비용 | 재설계 권장 |

---

## 옵션 성격과 비용의 관계

| 성격 | 일반적 c_eff | 특징 |
|------|-------------|------|
| `[S]` Safe | 0-5 | 낮은 리스크, 높은 reversibility |
| `[B]` Balanced | 4-8 | 적절한 균형 |
| `[I]` Ideal | 7-15+ | 높은 노력, 높은 불확실성 가능 |

---

## 계산 예시

### 예시 1: REST vs WebSocket vs gRPC

```
REST API (Safe):
  effort=1, risk=1, uncertainty=1, reversibility=5, resource=1
  c_eff = (1×1.0) + (1×1.5) + (1×1.2) - (5×1.0) - 1 = -2.3

WebSocket (Balanced):
  effort=2, risk=2, uncertainty=2, reversibility=4, resource=2
  c_eff = (2×1.0) + (2×1.5) + (2×1.2) - (4×1.0) - 2 = 1.4

gRPC (Ideal):
  effort=4, risk=3, uncertainty=3, reversibility=2, resource=3
  c_eff = (4×1.0) + (3×1.5) + (3×1.2) - (2×1.0) - 3 = 7.1

결론: REST가 최저 비용이지만, 요구사항에 따라 WebSocket도 적절
```

### 예시 2: 리팩토링 범위 결정

```
소규모 수정 (Safe):
  effort=1, risk=1, uncertainty=1, reversibility=5, resource=1
  c_eff = -2.3

모듈 분리 (Balanced):
  effort=3, risk=2, uncertainty=2, reversibility=3, resource=2
  c_eff = 3.9

전체 재구조화 (Ideal):
  effort=5, risk=4, uncertainty=4, reversibility=1, resource=4
  c_eff = 12.3

결론: 전체 재구조화는 c_eff=12.3으로 고비용 영역, 대안 탐색 권장
```

### 금지 사항

- 단순 장단점 나열만 하고 정량적 비교 없음 ❌
- c_eff 계산 없이 "권장" 표시 ❌
- 임계값 해석 누락 ❌
