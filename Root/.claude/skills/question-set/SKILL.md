# 질문셋 템플릿 스킬

새 기능 요청이나 요구사항 정리 시 구조화된 질문셋을 만듭니다.

## 언제 사용하나요?

- 새 기능 요청을 받았을 때
- 요구사항이 불명확할 때
- 스펙을 확정해야 할 때

## 질문셋 템플릿

```markdown
# 📋 [기능명] - 스펙 확정 질문셋

## 분석 축 (Analysis Axes)

이 결정의 핵심 축:
1. [축1명] (1-5)
2. [축2명] (1-5)

## 1. [카테고리명]

**Q1-1. [구체적인 질문]**
- A) [S] [옵션 1] - 안전한 선택
- B) [B] [옵션 2] ⭐ (권장) - 균형 잡힌 선택
- C) [I] [옵션 3] - 이상적이지만 고비용

**축 기반 평가:**
| 옵션 | 성격 | [축1] | [축2] | 종합 |
|------|------|-------|-------|------|
| A    | Safe | 1     | 2     | 3    |
| B    | Balanced | 3  | 3     | 6    |
| C    | Ideal | 5    | 4     | 9    |

**권장 이유:** [왜 특정 옵션을 권장하는지 - 축 기반 설명]

## 2. [다음 카테고리]

**Q2-1. [질문]**
...
```

## 옵션 성격 분류 (필수)

| 성격 | 표시 | 특징 |
|------|------|------|
| **Safe** | `[S]` | 리스크 최소, 보수적, 검증된 방식 |
| **Balanced** | `[B]` | 적절한 리스크/이득 균형 |
| **Ideal** | `[I]` | 최선의 결과, 고비용/고위험 가능 |

## 핵심 원칙

1. **실행 전에 충분히 계획하세요**
2. **계획 전에 충분히 질문하세요**
3. **권장 사양에 ⭐ 표시하세요**

## 질문 카테고리 예시

- 입력 파라미터
- 동작 방식
- 출력 형태
- 에러 처리
- 성능 요구사항
- UI/UX

## 완료 후

질문셋 답변이 확정되면 `docs/tasks/<task-id>.md`에 스펙 문서로 정리합니다.

## 형식 필수 적용 (강제)

> **⚠️ 이 스킬이 트리거되면 반드시 위 템플릿 형식을 사용해야 합니다.**

### 체크리스트

| # | 항목 | 필수 |
|---|------|------|
| 1 | 분석 축 정의 (1~3개) | ✅ |
| 2 | 질문 번호: Q{카테고리}-{순번} 형식 | ✅ |
| 3 | 옵션: A), B), C) 형식 | ✅ |
| 4 | **옵션 성격: [S], [B], [I] 표시** | ✅ |
| 5 | 권장 표시: ⭐ (권장) | ✅ |
| 6 | **축 기반 평가 테이블** | ✅ |
| 7 | 권장 이유: 축 기반 설명 | ✅ |

### 금지 사항

- 번호 없이 질문 나열 ❌
- 1/2/3 대신 반드시 A/B/C ❌
- **옵션 성격 [S]/[B]/[I] 누락 ❌**
- 권장 표시 누락 ❌
- 권장 이유 생략 ❌
- 자유 형식 사용 ❌
- **분석 축 없이 장단점만 나열 ❌**
