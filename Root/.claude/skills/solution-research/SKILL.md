# 솔루션 리서치 스킬 (Solution Research)

새로운 기능 구현 전, 이미 존재하는 좋은 솔루션이 있는지 검색합니다.

> **목적**: AI가 직접 구현하기 전에 "누군가 이미 해결했나?" 확인

---

## 언제 사용하나요?

### 자동 트리거
- `full-feature` 워크플로우 실행 시 (Planner 단계에서)

### 수동 트리거
- 사용자가 "리서치 해줘", "기존 솔루션 찾아줘" 요청 시
- 사용자가 `/workflow full-feature` 실행 시

### 사용하지 않는 경우
- 버그 수정, 리팩토링
- 기존 패키지/코드 확장
- 단순 기능 추가
- quick-fix 워크플로우

---

## 검색 범위 가이드

| 프로젝트 유형 | 우선순위 |
|--------------|---------|
| **Unity** | Unity Asset Store > UPM > GitHub > NuGet |
| **일반 C#** | NuGet > GitHub > Stack Overflow |
| **알고리즘** | 논문/블로그 > GitHub |
| **게임 디자인** | GDC/게임 분석 > GitHub |

### Unity 검색 키워드

```
검색 형식: "[기능] unity package/asset/library"

예시:
- "state machine unity package"
- "object pooling unity asset"
- "tween animation unity library"
- "network sync unity"
```

### 검색 소스

| 소스 | URL 패턴 | 용도 |
|------|----------|------|
| Unity Asset Store | assetstore.unity.com | 상용/무료 에셋 |
| OpenUPM | openupm.com | 오픈소스 패키지 |
| GitHub | github.com/topics/unity | 오픈소스 |
| NuGet | nuget.org | .NET 라이브러리 |

## 출력 형식

### 스펙 문서 내 "기존 솔루션 검토" 섹션

```markdown
## 0. 기존 솔루션 검토 (Research)

### 검색 키워드
- `[검색어 1]`
- `[검색어 2]`

### 발견된 솔루션

| 솔루션 | 출처 | 라이선스 | 상태 | 적합도 |
|--------|------|----------|------|--------|
| [이름] | [Asset Store/GitHub/NuGet] | [MIT/상용/etc] | [활발/유지보수중/비활성] | ⭐⭐⭐ |
| [이름] | ... | ... | ... | ⭐⭐ |
| 직접 구현 | - | - | - | ⭐⭐ |

### 솔루션별 분석

**[솔루션 1]**
- 장점: [...]
- 단점: [...]
- 비용: [무료/유료 $XX]
- 학습곡선: [낮음/중간/높음]

**[솔루션 2]**
- ...

**직접 구현**
- 장점: 완전한 제어, 의존성 없음
- 단점: 개발 시간, 검증 필요
- 예상 effort: [1-5]

### 권장

**[권장 솔루션]** - [권장 이유]

또는

**직접 구현 권장** - [기존 솔루션이 부적합한 이유]
```

## 평가 기준

### 적합도 평가 (⭐ 1-3)

| 기준 | 가중치 | 설명 |
|------|--------|------|
| 기능 일치도 | 높음 | 요구사항을 얼마나 충족하는가 |
| 유지보수 상태 | 높음 | 최근 업데이트, 이슈 대응 |
| 커뮤니티 | 중간 | 사용자 수, 문서화 |
| 라이선스 | 중간 | 프로젝트에 적합한가 |
| 학습곡선 | 낮음 | 팀이 빠르게 적응 가능한가 |

### 적합도 등급

| 등급 | 의미 |
|------|------|
| ⭐⭐⭐ | 강력 추천 - 요구사항에 잘 맞음 |
| ⭐⭐ | 고려 가능 - 일부 조정 필요 |
| ⭐ | 부분 활용 - 참고용 |

---

## 직접 구현 권장 시점

다음 경우 "직접 구현"이 더 나을 수 있음:

| 상황 | 이유 |
|------|------|
| 매우 단순한 기능 | 라이브러리 오버헤드 > 직접 구현 |
| 기존 솔루션이 과도하게 복잡 | 필요한 건 일부인데 전체를 가져와야 함 |
| 라이선스 문제 | 상용 프로젝트에 부적합 |
| 프로젝트 특수 요구사항 | 기존 솔루션으로 커버 불가 |
| 핵심 차별화 기능 | 외부 의존 없이 완전 제어 필요 |

## docs/research 분리 기준

### 스펙 문서에 포함 (기본)
- 일회성 리서치
- 해당 태스크에만 관련
- 결정 후 재참조 불필요

### docs/research/<topic>.md로 분리
- 범용적 주제 (상태머신, 네트워크 동기화 등)
- 여러 태스크에서 재사용 가능
- 팀 지식으로 축적 가치 있음

```
docs/research/
├── state-machine-solutions.md
├── network-sync-options.md
└── ui-framework-comparison.md
```
