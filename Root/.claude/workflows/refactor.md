# Refactor Workflow

리팩토링 작업을 위한 워크플로우입니다. Critic이 선행하여 리스크를 먼저 분석합니다.

## 흐름

```
critic → architect → executor → reviewer
```

## 사용 조건

| # | 상황 |
|---|------|
| 1 | 기존 코드 구조 변경 |
| 2 | 성능 최적화 |
| 3 | 기술 부채 해소 |
| 4 | 패턴 통일/개선 |

## 단계별 상세

### 1. Critic (필수)

**목적:** 리팩토링 필요성 및 리스크 분석

**분석 항목:**
- 현재 상태의 문제점
- 리팩토링 범위
- 잠재적 리스크
- 대안 검토

**산출물:**
- `docs/critic/<task-id>.md`

**핵심 질문:**
- 정말 필요한가?
- 더 작은 범위로 가능한가?
- 롤백 계획은?

**다음 단계:** architect

---

### 2. Architect (필수)

**목적:** 리팩토링 설계

**설계 항목:**
- Before/After 구조
- 마이그레이션 순서
- 영향 범위
- 테스트 전략

**산출물:**
- `docs/design/<task-id>.md`

**다음 단계:** executor

---

### 3. Executor (필수)

**목적:** 리팩토링 실행

**주의사항:**
- 작은 단위로 커밋
- 각 단계마다 테스트
- 롤백 가능한 상태 유지

**산출물:**
- 리팩토링된 코드
- 테스트 코드

**다음 단계:** reviewer

---

### 4. Reviewer (필수)

**목적:** 리팩토링 검증

**검증 항목:**
- 기능 동작 유지
- 성능 영향
- 코드 품질 개선

**산출물:**
- `docs/reviews/<task-id>.md`
- `docs/retro/<task-id>.md` (학습 기록)

---

## 예상 소요

4-6단계 (규모에 따라)

## 리스크 관리

| # | 리스크 | 대응 |
|---|--------|------|
| 1 | 범위 확대 | critic 재호출 |
| 2 | 테스트 실패 | 롤백 후 재설계 |
| 3 | 성능 저하 | 벤치마크 후 판단 |
