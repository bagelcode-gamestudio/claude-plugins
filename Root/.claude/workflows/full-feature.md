# Full Feature Workflow

새로운 기능을 처음부터 끝까지 개발하는 전체 워크플로우입니다.

## 흐름

```
planner → critic → architect → critic → executor → reviewer
```

## 🚀 병렬화 전략

### 병렬 실행 구간

```
[순차 필수]     [병렬 가능]              [순차 필수]
planner → critic#1 → ┬─ architect(client) ─┬→ critic#2 → executor → reviewer
                     └─ architect(server) ─┘
```

### 병렬 호출 예시

**Architect 단계에서 병렬화:**
```markdown
단일 메시지에서 Task tool 다중 호출:
- Task(architect): "클라이언트 측 구조 설계"
- Task(architect): "서버 측 구조 설계"
- Task(architect): "게임 엔진 측 설계" (해당시)
```

**Executor 단계에서 병렬화:**
```markdown
독립적인 구현 작업 분리:
- Task(executor): "클라이언트 코드 구현"
- Task(executor): "서버 API 구현"
```

### 병렬화 판단 기준

| 조건 | 병렬 가능 | 예시 |
|------|----------|------|
| 서로 다른 레이어 | ✅ | client vs server vs games |
| 같은 파일 수정 | ❌ | 충돌 위험 |
| 타입 의존성 없음 | ✅ | 독립적인 기능 |
| 결과가 다음 입력 | ❌ | planner → architect |

## DAG 구조 (Directed Acyclic Graph)

> **💡 연구 기반**: 계획을 직선이 아닌 그래프로 구조화하면 이해-기억-평가-순서가 모두 향상됩니다.

```
        ┌─────────────────────────────────────────────────────┐
        │                                                     │
        ▼                                                     │
    [M0: 요청]                                                │
        │                                                     │
        ▼                                                     │
    [M1: 스펙 확정] ◄───────────────────────────┐             │
        │                                       │             │
        ▼                                       │             │
    ┌──────────┐    risk high?                  │             │
    │ critic#1 │ ──────────────────────────────►┘             │
    └──────────┘                                              │
        │ ok                                                  │
        ▼                                                     │
    [M2: 설계 완료] ◄───────────────────────────┐             │
        │                                       │             │
        ▼                                       │             │
    ┌──────────┐    설계 문제?                   │             │
    │ critic#2 │ ──────────────────────────────►┘             │
    └──────────┘                                              │
        │ ok                                                  │
        ▼                                                     │
    [M3: 구현 완료] ◄───────────────────────────┐             │
        │                                       │             │
        ▼                                       │             │
    ┌──────────┐    반려?                       │             │
    │ reviewer │ ──────────────────────────────►┘             │
    └──────────┘                                              │
        │ 승인                                                │
        ▼                                                     │
    [M4: 검증 완료]                                           │
        │                                                     │
        ▼                                                     │
    [M5: 머지] ◄──────────── 핫픽스 필요 시 ──────────────────┘
```

---

## Milestone 정의

| Milestone | 완료 조건 | 산출물 | 실패 시 |
|-----------|----------|--------|--------|
| **M0** | 사용자 요청 접수 | - | - |
| **M1** | 스펙 문서 승인 | `docs/tasks/<task-id>.md` | 질문셋 재작성 |
| **M2** | 설계 문서 승인 | `docs/design/<task-id>.md` | 설계 재검토 |
| **M3** | 코드 구현 + 테스트 통과 | 소스 코드 | 버그 수정 |
| **M4** | 리뷰 승인 | `docs/reviews/<task-id>.md` | 수정 후 재리뷰 |
| **M5** | main 머지 | 머지 커밋 | 충돌 해결 |

---

## 분기 조건 (Transition Cost)

각 분기점에서 비용 평가를 수행합니다.

| 노드 | 분기 조건 | c_eff 임계값 | 분기 결과 |
|------|-----------|-------------|----------|
| Critic #1 | 스펙 리스크 | c_eff > 8 | M1으로 회귀 |
| Critic #2 | 설계 리스크 | c_eff > 8 | M2로 회귀 |
| Reviewer | 품질 미달 | - | M3로 회귀 |

## 단계별 상세

### 0. Solution Research (full-feature 전용)

**목적:** 기존 솔루션/라이브러리 검색 ("바퀴 재발명" 방지)

**동작:**
- WebSearch로 기존 솔루션 탐색
- Unity Asset Store, GitHub, NuGet 등 검색
- 솔루션별 적합도 평가

**산출물:**
- 스펙 문서 내 "기존 솔루션 검토" 섹션

**스킬:** `solution-research`

**완료 조건:**
- 최소 2개 소스 검색
- 솔루션 vs 직접 구현 비교 완료

**다음 단계:** planner (질문셋 작성)

---

### 1. Planner (필수)

**목적:** 요구사항 정리, 질문셋 작성 (+ 기존 솔루션 검토 결과 반영)

**산출물:**
- 질문셋 문서 (기존 솔루션 검토 섹션 포함)
- `docs/tasks/<task-id>.md` 스펙 문서

**완료 조건:**
- 사용자가 모든 질문에 답변
- 스펙 문서 승인

**다음 단계:** critic

---

### 2. Critic #1 (필수)

**목적:** 질문셋/스펙 검토

**검토 항목:**
- 누락된 중요 질문
- 편향된 옵션
- 장기 리스크

**산출물:**
- 검토 의견 (인라인)

**완료 조건:**
- 검토 완료, 필요시 planner 재수행

**다음 단계:** architect

---

### 3. Architect (필수)

**목적:** 설계 문서 작성

**산출물:**
- `docs/design/<task-id>.md` 설계 문서

**완료 조건:**
- 설계 문서 작성 완료

**다음 단계:** critic

---

### 4. Critic #2 (필수)

**목적:** 설계 검토

**검토 항목:**
- 더 간단한 대안
- 기술 부채 누적
- 유지보수 용이성

**산출물:**
- `docs/critic/<task-id>.md` (선택)

**완료 조건:**
- 설계 승인

**다음 단계:** executor

---

### 5. Executor (필수)

**목적:** 구현

**에이전트:** `executor` (코드/씬/프리팹 모두 담당)

**산출물:**
- 구현된 코드
- 테스트 코드
- 커밋 히스토리

**완료 조건:**
- 구현 완료
- 테스트 통과

**다음 단계:** reviewer

---

### 6. Reviewer (필수)

**목적:** 품질 검증, 리뷰

**검토 항목:**
- 코드 품질
- Unity 검증 (해당시)
- 테스트 확인

**산출물:**
- `docs/reviews/<task-id>.md`
- (필요시) `docs/retro/<task-id>.md`

**완료 조건:**
- 승인 또는 반려

|------|-----------|------|
| planner | ❌ | - |
| critic #1 | ⚠️ | 매우 간단한 기능 |
| architect | ❌ | - |
| critic #2 | ⚠️ | 영향 범위 작음 |
| executor | ❌ | - |
| reviewer | ❌ | - |
